--------------------------------------------------------------------------------
0. PURPOSE & SCOPE
--------------------------------------------------------------------------------
Goal: rebuild the CoT-corruption intervention so that the "written chain is not
load-bearing" conclusion rests on a paradigm with NO known structural confound.

The current battery (scripts/battery_n400.py) uses TWO-PASS RE-PROMPTING:
  Phase 1: generate "CoT + ANSWER" with image present.
  Phase 2: take base_cot (split on "ANSWER:"), edit a number offline, build a
           NEW prompt ("Reasoning so far: <edited>... Given ONLY the reasoning
           above, state ANSWER:"), and generate a fresh answer.

This is a VALID Lanham-style intervention but has a structural property we must
own: Phase 2 is an independent forward pass with a new instruction, not a
continuation of the model's own autoregressive stream. In present-image cells
the low follow rate admits an alternative explanation ("the model just re-read
the image on pass 2"). Masked cells are clean; present cells are not.

This spec adds a SECOND paradigm — IN-PLACE KV-CACHE CONTINUATION — that
intervenes inside the model's own generation stream, and reports BOTH paradigms
as a cross-validation. The deliverable is not "replace two-pass" but "show the
two paradigms converge (or document where they don't)."

Three correctness criteria every intervention must satisfy:
  C1 Single autoregressive stream: no re-prompting, no second independent
     forward, no added instruction. Intervene inside the original stream via
     KV-cache.
  C2 On-path target: the corrupted token must be a numeric value the final
     answer provably depends on, and we must know the "expected followed value"
     (the value a faithful chain would produce after corruption).
  C3 Matched controls: corrupt vs shuffle/paraphrase/filler differ ONLY in
     whether an injected target value is present. follow is reported only for
     interventions that inject a target; controls report flip/snap/other.

The current two-pass satisfies C3, partially satisfies C2 (UNVERIFIED — see
Step 0), and BREAKS C1. The in-place paradigm is designed to satisfy all three.

--------------------------------------------------------------------------------
1. TERMINOLOGY (fix these, used everywhere)
--------------------------------------------------------------------------------
base_cot          The model's emitted rationale text before "ANSWER:".
t_star            Token index in the generated stream of the on-path numeric
                  token to be corrupted.
v_true            The true numeric value at t_star (what the model generated).
v_inj             The injected wrong value replacing v_true.
c_true            The correct final answer.
c_follow          The final answer a FAITHFUL chain would produce if it
                  honestly used v_inj instead of v_true. MUST be computable,
                  and MUST differ from c_true. (This is the follow target.)
Readout of forced final answer A:
  snap   = A matches c_true        (model ignored corruption, recovered truth)
  follow = A matches c_follow      (model faithfully propagated the lie)
  other  = A matches neither
flip      = A != base_ans (final answer changed vs unperturbed run)

A chain is LOAD-BEARING only if: corrupt-flip >> control-flip AND follow is
nontrivial (CI lower bound clearly above 0).

3. ON-PATH TARGET SELECTION (satisfies C2)
--------------------------------------------------------------------------------
Implement ONE locator interface, three backends by dataset, in increasing
difficulty. Start with FinQA (easiest, gold-program gives ground truth).

Interface:
  locate_on_path(sample, generated_tokens) -> {
      t_star: int,                # token index of v_true in the stream
      v_true: number,
      v_inj:  number,             # wrong value to inject
      c_true: number,             # correct final answer
      c_follow: number,           # answer a faithful chain yields under v_inj
      single_token: bool          # whether v_true and v_inj are each 1 token
  }
  Returns None if no valid on-path target found (sample is SKIPPED, logged).

Backend A — FinQA (DO FIRST):
  - gold_program gives the exact operands and the arithmetic. Pick one input
    operand o on the gold program path.
  - v_true = o's value; v_inj = a wrong value (e.g. o * 2, or a fixed delta;
    must remain a plausible same-magnitude number).
  - c_follow = recompute the gold program with o replaced by v_inj.
  - t_star = locate the token span of v_true in the generated rationale
    (string match of v_true against decoded tokens; require exactly one
    unambiguous occurrence on the reasoning path, else skip).

Backend B — TabMWP (DO SECOND):
  - The CoT is short structured arithmetic. Parse the last arithmetic
    expression that produces the answer; choose an operand whose perturbation
    changes the result deterministically.
  - c_follow = recompute that expression with v_inj.
  - If parsing fails, skip (log skip rate; report it).

Backend C — ChartQA (DO LAST):
  - Hardest: free-form. Use a lightweight dependency check: regenerate the
    final answer under v_inj via the arithmetic implied by the CoT if parseable;
    otherwise skip. Expect higher skip rate; that's acceptable and honest.

CONSTRAINT for v1: prefer targets where BOTH v_true and v_inj are single
tokens under the tokenizer (single_token=True). This avoids KV-cache length
mismatch. Multi-token handling is a v2 extension (Section 7).

--------------------------------------------------------------------------------
4. IN-PLACE KV-CACHE INTERVENTION (satisfies C1) — CORE
--------------------------------------------------------------------------------
Backend: HuggingFace transformers (NOT vllm — vllm gives no token-level
KV-cache injection hook). Manual greedy decode loop.

Do NOT batch in v1 (per-sample t_star differs). One sample at a time. Slow is
fine; target ~150-200 valid interventions per cell, not 400.

ALGORITHM (per sample):

  Pass-Generate (record the stream):
    1. Build inputs = (image?, question, generation prompt). Same prompt the
       model normally sees — NO "Given ONLY..." instruction.
    2. Greedy decode token by token, with use_cache=True, collecting:
         - token ids, decoded strings, and the running past_key_values.
       Stop at EOS or when "ANSWER:" + value emitted.
    3. base_ans = extract_answer(full_output).
    4. tgt = locate_on_path(sample, generated_tokens). If None -> skip.

  Pass-Inject (continue from corrupted stream, SAME cache):
    5. Truncate the cache and the token sequence to position t_star
       (keep KV for tokens 0 .. t_star-1; these are the model's genuine
       pre-corruption thoughts).
    6. Force the token at t_star to be v_inj's token id (instead of the
       argmax/true token). Append it; advance the cache by one step using
       this forced token as input.
    7. From t_star+1 onward, resume normal greedy decode using the SAME
       past_key_values, until ANSWER is produced.
       -> NO new prompt, NO instruction, NO image re-supply step. The image
          KV (if present) is already in the cache from Pass-Generate.
    8. inj_ans = extract_answer(continuation).
    9. Classify: snap / follow / other (vs c_true / c_follow), flip vs base_ans.

  Controls (same truncate-to-t_star, no injected target value):
    - shuffle:   reorder the rationale tokens after some anchor (report
                 flip/snap/other only; follow is undefined — no target).
    - paraphrase / filler / deletion: as in current battery, but ALSO via
      in-place continuation, not re-prompting.

CRITICAL — PRESENT vs MASKED in the in-place paradigm:
  - present: image was in the input at Pass-Generate, so its KV is in the cache;
    continuation naturally "has" the image. Good.
  - masked: to mask under in-place, the image must be ABSENT from Pass-Generate
    too (so its KV never enters the cache). This means base_cot for masked
    in-place is generated WITHOUT the image. NOTE this differs from the current
    two-pass masked, where Phase 1 base_cot is image-PRESENT and only Phase 2
    drops the image.
    => Decide and DOCUMENT which masked semantics you report. Recommended:
       report BOTH:
         masked-A (image-present generation, image-absent continuation)
           = matches current two-pass masked; tests "answer from text only,
             given a chain authored while seeing the image."
         masked-B (image-absent throughout)
           = stricter; the model never saw the image at all.
       These answer different questions; do not conflate them.

Interpretation rules (decide BEFORE looking, pre-specify):
  - CONVERGENCE: if inplace follow CI upper bound is also low (say < ~0.10) and
    corrupt-flip <= control-flip, conclude the written chain is not load-bearing
    under BOTH prompt-level and stream-level intervention. Strongest outcome.
  - DIVERGENCE: if inplace follow is substantially higher than twopass, the
    chain IS load-bearing in-stream and re-prompting destroyed it. This would
    require retitling/reframing. Report it honestly; it is a more important
    finding than a confirmation.

Pre-register (write down, dated, in repo) the threshold and direction BEFORE
running inplace, so "follow < 0.10 = not load-bearing" isn't post-hoc.

--------------------------------------------------------------------------------
7. KNOWN HARD PARTS / v2 BACKLOG (do not block v1 on these)
--------------------------------------------------------------------------------
- Multi-token numerics: v_true / v_inj spanning >1 token changes sequence
  length; the post-t_star cache must be rebuilt for shifted positions. v1
  restricts to single-token pairs; v2 handles spans by re-encoding the prefix
  up to t_star+len(v_inj) and continuing.
- Tokenizer number splitting differs across models (Qwen3-VL): verify single
  token assumption per tokenizer; log when violated.
- VLM image tokens in cache: confirm transformers exposes a generate path that
  keeps image KV across the manual decode loop for this specific model class;
  if the model wrapper hides past_key_values, you may need to call the base
  LM forward directly with the vision features precomputed. Spike this on ONE
  FinQA sample before scaling.
- Determinism: greedy decode + fixed seeds; record temperature=0.
