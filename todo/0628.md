## 0.5 研究问题重定义：Numeric Path-Specific Causal Audit

**Spec**

新的核心问题不是泛泛地问 “VLM 是否 reasoning”，也不是只问 “emitted CoT 是否 faithful”。更精确的研究问题是：

> 在 chart/table 数值任务中，最终答案的因果路径是主要经过 localized visual numeric source，经过 textualized CoT number，还是二者在不同阶段分工？

操作性路径：

1. `visual source -> CoT number`：图表/表格中某个 cell/bar/point/region 的数值改变后，模型写出的对应 CoT 数字是否跟着改变。
2. `CoT number -> answer`：同一自回归流中强制 CoT 数字为 counterfactual value 后，最终答案是否 follow。
3. `visual source -> answer | CoT held fixed`：视觉源改变但 CoT 数字被强制保持原值时，最终答案是否仍被视觉反事实拉走。
4. conflict arbitration：视觉源和文字链冲突时，答案跟视觉、跟文字，还是输出第三类。

这个 framing 对应 `causal domain decomposition of numerical VLM reasoning`。它和 2026-06 前后已有工作区分如下：

- 不声称 first to study visual-vs-textual reasoning。Att-CoT、TVI-CoT、VEPA、ViGOS、VIGIL、VISE、Composer、EVA 等已经把高层问题打得很近。
- 不把 attention/saliency 当作主因果证据。attention/attribution 只可作为辅助定位或 appendix diagnostic。
- 不只做 grounded CoT 训练或 post-hoc attribution。本文主张是 source-specific、numeric、counterfactual、path-specific 的因果审计。

**Steps**

- [ ] 在 prereg 中写明 primary claim：
  `We estimate path-specific causal effects for numerical VLM reasoning by independently intervening on localized visual sources and textualized intermediate numbers.`
- [ ] 定义论文里的三个术语：
  `visual source`、`textualized numeric intermediate`、`final numeric answer`。
- [ ] 定义不审计的对象：
  `not audited = all hidden-state latent computation not mediated by the chosen source/token path`。
- [ ] 在 related work/positioning table 中把本文和以下工作区分：
  visual dependence during CoT、grounded CoT training、visual token attention/attribution、latent visual thought tokens、chart answer attribution。

**Acceptance**

- [ ] 论文 claim 不再写成 “where does VLM reasoning happen?” 的宽泛命题，而写成 chart/table numeric path-specific causal effects。
- [ ] reviewer 不能用已有 visual-dependence / grounded-CoT 工作一句话判定撞题；差异能落到 intervention object 和 path-specific readout。

## 0.6 四格冲突实验设计

**Spec**

主实验使用 2x2 conflict design。每个 valid attempt 都围绕同一个 on-path numeric source 构造：

| visual source | emitted / forced CoT number | cell name | 目的 |
|---|---|---|---|
| true | true | `V_true_T_true` | base sanity |
| false | false | `V_cf_T_cf` | coherence check：视觉和文字一致为反事实 |
| false | true | `V_cf_T_true` | 测 `visual source -> answer` 是否能越过固定文字链 |
| true | false | `V_true_T_cf` | 测 `textualized number -> answer` 是否能越过真实视觉证据 |

其中：

- `v_true`：原始视觉源和原始 CoT 中的数值。
- `v_cf` / `v_inj`：同一 source 的 counterfactual value。
- `c_true`：原始 gold program / parsed expression 的最终答案。
- `c_cf`：把 source value 从 `v_true` 改成 `v_cf` 后重算的最终答案。
- `T_cf` 优先通过 in-place KV force 实现；不能 re-prompt。
- `V_cf` 优先通过 counterfactual rendering/editing 实现；不能只改 prompt 文本。

主读数：

- `E_visual_to_text = P(CoT_number = v_cf | V_cf) - P(CoT_number = v_cf | V_true)`。
- `E_text_to_answer = P(answer = c_cf | V_true, T_cf) - P(answer = c_cf | V_true, T_true)`。
- `E_visual_direct_to_answer = P(answer = c_cf | V_cf, T_true) - P(answer = c_cf | V_true, T_true)`。
- `conflict_winner`：
  `visual` if answer matches `c_cf` under `V_cf_T_true` or `V_cf_T_conflict`；
  `text` if answer matches text-implied answer under `V_true_T_cf`；
  `snap_true` if answer matches original `c_true`；
  `other` otherwise。

**Steps**

- [ ] 扩展输出 schema，新增字段：
  `path_cell, source_id, source_type, source_region, source_surface, source_true, source_cf, text_true, text_forced, c_source_true, c_source_cf, answer_source, answer_text, conflict_winner`。
- [ ] 新增 `VisualSourceTarget` dataclass：
  `sample_id, source_id, source_type, region, v_true, v_cf, c_true, c_cf, renderer_backend, evidence`。
- [ ] 新增 `PathProbeRow` 或扩展 `ProbeRow`，要求每个四格 cell 可用 `sample_id + target_id + path_cell` join。
- [ ] 定义 `answer_from_source_cf` 与 `answer_from_text_cf` 的 grading helper；当二者相同时跳过 conflict arbitration 样本。
- [ ] 每个 dataset backend 必须声明：
  source 可定位性、source 可编辑性、`c_cf` 可重算性、CoT token 可定位性。
- [ ] prereg 主 cell：
  v1 先跑 FinQA rendered table / synthetic table；v1.5 再跑 TabMWP；v2 扩到 chart/table/counting/map。

**Acceptance**

- [ ] 每个进入主表的样本至少能填满 `V_true_T_true`、`V_true_T_cf`、`V_cf_T_true` 三格；`V_cf_T_cf` 可作为 coherence check，但不能替代冲突格。
- [ ] `c_cf` 永远来自 source-level recompute，不是 injected operand 本身。
- [ ] `visual_direct` 和 `text_mediated` 两个 effect 可以在同一批 sample 上 paired 比较。

## 0.7 任务/数据集选择原则

**Spec**

数据集不按名气选，而按是否支持 path-specific causal audit 选。每个候选任务先打五个分：

1. `source_localizable`：能否定位到 cell/bar/point/region/object set。
2. `source_editable`：能否生成无 artifact 的 counterfactual visual input。
3. `answer_recomputable`：改 source 后能否确定 `c_cf`。
4. `cot_token_locatable`：模型是否会写出可定位的 numeric intermediate。
5. `visual_realism`：真实感和外部效度。

推荐路线：

| tier | dataset/task | source type | reason | risk |
|---|---|---|---|---|
| v1 | FinQA rendered table | table cell | gold program、source cell 可控、`c_cf` 可重算 | 任务偏金融表格，覆盖窄 |
| v1 | synthetic rendered table | table cell | counterfactual render 最干净，适合 spike | synthetic 真实性弱 |
| v1.5 | TabMWP | table cell / expression operand | 贴近当前 headline table arithmetic | gold program 弱，parse-and-skip |
| v2 | TAT-QA table-only | financial table cell | derivation/scale 强，真实 report | text evidence 混杂，需要筛 table-dominant |
| v2 | PlotQA / DVQA / FigureQA / LEAF-QA | bar/line/point | chart source 可重渲染，metadata 强 | 图表编辑和 OCR/tick confounds |
| v2 | HiTab / MultiHiertt | hierarchical table cell | 层级索引和真实金融表更强 | source alignment 和 rendering 难 |
| v3 | MapQA / MapVerse | region color + legend | 测 region/legend 数值路径，不只表格/图表 | legend-color mapping 容易有 confound |
| v3 | TallyQA / HowMany-QA | object set/count | 对照“视觉计数”而非 printed number reading | `c_cf` 需图像编辑或 synthetic scene |
| v3 | CLEVR / GQA | scene graph objects/counts | functional program 和 counterfactual 最干净 | synthetic/general VQA，离 paper 主线远 |
| v3 | GeoQA / Geometry3K / MathVista / MathVerse / DynaMath | diagram number/relation | 测 geometry/diagram 数值推理 | source edit 和 derivation path 难 |
| v3 | SPIQA / SciGraphQA / SciTabQA / SCITAT / SciNumQA | scientific figure/table | external validity 强 | 多数样本缺可执行 derivation，需高 skip |

**Steps**

- [ ] 新增 `docs/snapshots/0628_dataset_selection.md`，逐个记录候选数据集的五项评分、样例、source edit 策略和 skip 风险。
- [ ] v1 只允许选择 source edit 最干净的 table/cell substrate，不从 ChartQA free-form 起步。
- [ ] 每扩一个数据集，先做 `n=30` manual/automatic audit：
  source 是否可定位、counterfactual render 是否自然、模型是否写目标数字、`c_cf` 是否可重算。
- [ ] 论文主线至少覆盖两个数值来源家族后再写 generalized claim：
  e.g. table cell + chart mark，或 table cell + counting object set。

**Acceptance**

- [ ] 数据集扩展不是“再跑一个 benchmark”，而是扩展一种 visual numeric source family。
- [ ] 每个数据集进入 v2/v3 前都有明确 go/no-go：valid rate、source edit quality、`c_cf` reliability。

### v1-2 Qwen3-VL Manual Greedy Decode Spike

**Spec**

v1 最大技术风险是 Qwen3-VL 的 multimodal cache 是否能在 HuggingFace manual decode 中稳定使用。先做一条 FinQA sample spike，不进入规模实验。

需要验证四件事：

1. manual greedy decode 与 `model.generate(... do_sample=False)` 在同一 prompt 下输出一致或足够接近。
2. prefill 时喂 image/table，后续 token-by-token decode 只喂 token + `past_key_values`，不再喂 image。
3. 在目标 token 前保存/clone prefix cache 后，force 原始 `v_true` token 能 replay 出与原始 stream 相同的 continuation。
4. force `v_inj` token 后，continuation 确实从同一 cache branch 继续，不出现 shape/cache_position 错误。

**Steps**

- [ ] 在 `app/eval_distill/inplace_kv.py` 实现 `build_qwen_inputs(processor, image, question, prompt)`，沿用当前脚本的 chat template。
- [ ] 实现 `manual_prefill(model, batch)`：
  调用 `model(..., use_cache=True)`，保存 logits 和 `past_key_values`。

### v1-3b FinQA Path-Specific Four-Cell Runner

**Spec**

在 v1-3 的 in-place text intervention 通过后，跑 FinQA path-specific 四格。主 cell 至少需要：

- `V_true_T_true`：原始 rendered table，原始 greedy CoT/answer。
- `V_true_T_cf`：原始 rendered table，在 CoT target token 处 force `v_cf`，继续生成。
- `V_cf_T_true`：counterfactual rendered table，但在 CoT target token 处 force 原始 `v_true`，继续生成。
- `V_cf_T_cf`：counterfactual rendered table，允许或 force CoT target 为 `v_cf`，作为 coherence check。

关键判读：

- 如果 `V_true_T_cf` 跟 `c_cf`，说明 textualized number 在原 stream 内 load-bearing。
- 如果 `V_cf_T_true` 跟 `c_cf`，说明最终答案仍可从 visual source 直接拉回 counterfactual source，即存在 `visual -> answer` direct path。
- 如果 `V_cf_T_true` 仍跟 `c_true`，说明固定文字链压过视觉反事实，答案形成更接近 text-mediated。
- 如果两格都不跟，可能是 conflict OOD 或 cache/renderer artifact，需要 controls。

**Steps**

- [ ] 实现 `scripts/probe_path_causal.py --dataset finqa`：
  读取同一个 target set，调度 `V_true/V_cf` 输入和 `T_true/T_cf` force policy。
- [ ] 对 `V_cf` 先完整 greedy decode，确认模型自然写出的 target number 是否更可能为 `v_cf`：
  这是 `E_visual_to_text` 的主读数。
- [ ] 对 `V_cf_T_true`：
  用 counterfactual image prefill/base generation，但在目标数字位置 force 原始 `v_true`；如果 `v_true` 在 counterfactual CoT 中没有自然出现，则从 matched target slot 或 first target numeric slot 分支，记录 `slot_policy`。
- [ ] 对 `V_true_T_cf`：
  复用 v1-3 in-place branch，从原始 image/table stream force `v_cf`。
- [ ] 对每个样本保存四格 row，并用 `target_id` join。
- [ ] 增加 sanity controls：
  `V_cf_T_cf` answer 应高比例匹配 `c_cf`；若不匹配，说明 renderer/source edit 或 base model perception 不可靠。
- [ ] 保存结果到：
  `data/distill/poc/path_finqa_8b_tablecell.jsonl`。

**Acceptance**

- [ ] 至少 100 个样本能完成 `V_true_T_true`、`V_true_T_cf`、`V_cf_T_true` 三格；目标 150-200。
- [ ] `V_cf_T_cf` coherence answer match rate 足够高，或 summary 明确说明视觉 counterfactual 不被模型读出，不能解释为 path effect。

- [ ] 若 `V_cf_T_true` 需要 slot-policy 对齐，论文中透明报告，不把它写成完全自然的 CoT slot。

## 3. v1.5 - TabMWP 主表格算术扩展

### v1.5-1 TabMWP On-Path Locator

**Spec**

v1.5 的目标是把 FinQA 的 clean result 扩展到论文 headline 的 table arithmetic substrate，并测试第二个 table-cell numeric source family。TabMWP 没有像 FinQA 一样直接可用的 gold program，因此 locator 采用 conservative parse-and-skip：

- 从模型 emitted CoT 中解析最后一个能产生 base/gold answer 的 arithmetic expression。
- 只接受安全表达式：数字、括号、`+ - * /`、百分号 normalization、简单等号链。
- 选择一个 operand，使 `operand -> v_inj` 后表达式重算结果 `c_follow` 与 `c_true` 不同。
- 如果 parse 或 dependency check 不确定，skip，不做 LLM 判定。
- visual source 必须能在 rendered table/text 中定位到唯一 cell 或题面数值；找不到 source provenance 的 expression operand 只能用于 `text -> answer` in-place，不进入完整 path-specific 主表。

**Steps**

- [ ] 实现 `app/eval_distill/onpath_tabmwp.py`。
- [ ] 实现 safe arithmetic parser：
  用 Python `ast` whitelist 或手写 parser；禁止 `eval` 任意字符串。
- [ ] 支持常见 CoT 模式：
  `a + b = c`、`a - b = c`、`a * b = c`、`a / b = c`、`(a+b)/c = d`、`Total = ...`。
- [ ] 从 CoT 后往前扫描 equation candidates，优先 result 匹配 `base_ans/gold` 的表达式。
- [ ] 对 candidate operands 计算 `v_inj` 和 `c_follow`，要求 `c_follow != c_true`。
- [ ] 实现 `source_tabmwp.py`：
  将 parsed operand 对齐到 table cell / problem field；若 operand 多处出现，skip `source_ambiguous`。
- [ ] 实现 TabMWP counterfactual renderer：
  修改目标 table cell / source field，并保存 `V_cf`。
- [ ] 复用 v1 token-span matching，v1.5 仍默认 single-token；multi-token 放 v2。
- [ ] 添加 TabMWP fixture tests：
  simple addition、subtraction、percentage、ambiguous expression、expression result not answer。
- [ ] dry-run 统计 `data/distill/tabmwp/test_cases_400.jsonl` 上的 parse rate 和 expected valid rate。

**Acceptance**

- [ ] TabMWP locator 宁可 skip，也不接受无法重算 `c_follow` 的样本。
- [ ] 每个 valid TabMWP target 的 `c_follow` 来自 expression recompute，不是 injected operand 本身。
- [ ] 每个进入 path-specific 主表的 TabMWP target 都有 source provenance；只有 on-path text target 而无 source 的样本只能进入 in-place appendix。
- [ ] dry-run 报告 parse/skip reason，能支撑论文中的 `n_raw -> n_probe` flow。

### v2-2 ChartQA / PlotQA / DVQA Conservative Source Locator

- expression result 必须匹配 base/gold answer。

- 对纯 visual lookup、yes/no、label answer、无表达式样本全部 skip。
- visual source 必须能定位到 chart mark / data point / table-like visual value / axis-derived value；如果只能定位 text mention 不能定位 visual mark，则降为 text-only appendix。

- [ ] 再 dry-run ChartQA 8B/32B base outputs，估算 parseable arithmetic + source-locatable subset。

### v2-3 Full In-Place Control Battery

**Spec**

v2 尝试把 `todo/06.md` 的 controls 全部迁移到 in-place branch：

- `corrupt`：force target value/token sequence。
- `local_filler`：local no-target disruption。
- `offpath_number`：非依赖数字替换。
- `forced_filler_suffix`：length-matched filler suffix。
- `forced_shuffle_suffix`：same-token shuffled suffix。

## 6. Go / No-Go Gates

**Spec**

**Spec**

**Steps**

- [ ] 若 80 <= valid n < 150：只进 appendix 或标注 underpowered，并优先推进 multi-token v2。

**Acceptance**

**Spec**

**Acceptance**

- [ ] 结果解释先服从实验，不反向服务标题。

### Gate D: Panel Pitfall Guardrails

**Spec**

每个 v1/v1.5/v2 结果包都必须通过从 `todo/0627.md` 提炼出的审稿护栏。它们不是写作阶段的附加项，而是结果能否进入主文的前置条件。

护栏覆盖：

- 方法透明度：reviewer 不读代码也能重建实验。
- 统计不确定性：不再只报裸比例。
- 样本选择偏差：报告 raw -> eligible -> valid 的流失，并做 sensitivity。
- control 清洁度：至少有一个温和、局部、no-target control，不只依赖 harsh shuffle。
- claim 边界：区分 `text-path`、`visual-source path`、`full path-specific`；不否定 latent computation。
- field positioning：明确不是重复 Lanham-style probe，而是 VLM distillation + fixed visual evidence + localized source editing + stream-level intervention。
- 机制语言：`snap` 不自动等于“模型视觉重读”；除非有视觉反事实，否则写成 "answer remains recoverable from visible evidence / original source"。
- 外部效度：若仍只用 Qwen-family，主文明确收窄。
- AAAI 合规：模板、双盲、reproducibility checklist、artifact appendix。

**Steps**

- [ ] 每个 clean probe summary 同时输出 `n_raw -> n_base_correct -> n_cot_present -> n_locator_valid -> n_single_or_multi_token_valid -> n_scored`。
- [ ] 每个 path-specific summary 同时输出 `n_raw -> n_source_localizable -> n_source_editable -> n_cf_recomputable -> n_conflict_identifiable -> n_scored`。
- [ ] 每个 follow/snap/flip rate 带 95% CI；`corrupt_flip - control_flip` 带 paired bootstrap 或 exact randomization CI。
- [ ] 每个 path effect 带 paired bootstrap CI；同一样本四格不齐时标注 unpaired/partial，不进主效果。
- [ ] 对 multi-cell headline 结果标明 confirmatory vs diagnostic；需要多重比较时用 Holm/BH，或明确不做 confirmatory claim。
- [ ] 对 skipped/failed locator 样本做 sensitivity bound：
  failed parse as non-follow / unknown 的上下界至少在 appendix 报告。
- [ ] 每个主 cell 至少有一个 no-target local/offpath control；若没有，只能进 appendix，不能作为 headline causal result。
- [ ] 对 stable-correct、SFT-gain、SFT-loss、both-wrong 至少在 TabMWP 或 FinQA 主 cell 报一个分层分析；若某层不适用，写明原因。
- [ ] 每个 result snapshot 写明：
  `object audited = localized visual source and/or emitted written rationale`，并分清具体 cell；`not audited = all other hidden-state / latent computation paths`。
- [ ] 文稿中把 TallyQA 标为 external regime check，FinQA 标为 constructive/gold-program operand test，不把异质 substrate 硬堆成同一个因果证明。
- [ ] 文稿中把 `re-read` 降调为 "snap to recoverable source"；只有完成局部遮挡或 counterfactual chart/table editing 后才恢复强 `re-read` 机制语言。
- [ ] 如果没有非 Qwen 复现，Abstract/Discussion/Limitations 明确写 "single model family" 或等价限定。
- [ ] 每次 paper edit 后检查：
  title/abstract/intro/result/discussion 中的 `not reasoning`、`does not compute`、`internalization failed` 等强措辞是否只在限定范围内出现。
- [ ] 在 Method 增加或维护定位表：
  text CoT / visual evidence / tool trace / latent state / chart-table QA；列为 object audited、intervention type、model setting、allowed conclusion。
- [ ] 在 Discussion 增加 deployment audit implications：

- [ ] AAAI 版本前完成：
  AAAI template/page limit、双盲 metadata/path/link 检查、reproducibility checklist、artifact appendix、最小复现命令。

**Acceptance**

- [ ] 任一结果进入主文前，都有方法透明度、CI、selection flow、control、claim-boundary 五项证据。
