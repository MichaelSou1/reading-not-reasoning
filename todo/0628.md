## 2. v1 - FinQA 最小决定性闭环

### v1-1 FinQA On-Path Locator

**Spec**

v1 只做 FinQA，因为它有 gold program，是最快满足 C2 的 substrate。目标不是扩大覆盖面，而是得到一个审稿人难以攻击的 clean cell：

- dataset：FinQA rendered table。
- model：Qwen3-VL 8B FinQA curriculum arm，优先 `data/distill/poc/lora_8b_finqa_b2`。
- condition：present image/table。
- target：gold-program flippable table operand。

- locate policy：`v_true` 必须在模型生成的 rationale 中作为唯一、无歧义 numeric token/span 出现；多次出现或只出现在 final answer 之后则 skip。

### v1-2 Qwen3-VL Manual Greedy Decode Spike

**Spec**

v1 最大技术风险是 Qwen3-VL 的 multimodal cache 是否能在 HuggingFace manual decode 中稳定使用。先做一条 FinQA sample spike，不进入规模实验。

需要验证四件事：

1. manual greedy decode 与 `model.generate(... do_sample=False)` 在同一 prompt 下输出一致或足够接近。
2. prefill 时喂 image/table，后续 token-by-token decode 只喂 token + `past_key_values`，不再喂 image。
3. 在目标 token 前保存/clone prefix cache 后，force 原始 `v_true` token 能 replay 出与原始 stream 相同的 continuation。
4. force `v_inj` token 后，continuation 确实从同一 cache branch 继续，不出现 shape/cache_position 错误。

**Steps**

- [ ] 在 `app/eval_distill/inplace_kv.py` 实现 `build_qwen_inputs(processor, image, question, prompt)`，沿用当前脚本的 chat template。
- [ ] 实现 `manual_prefill(model, batch)`：
  调用 `model(..., use_cache=True)`，保存 logits 和 `past_key_values`。
- [ ] 实现 `manual_next(model, token_id, past_key_values, attention_mask/cache_position)`：
  优先使用 `model.prepare_inputs_for_generation` 适配 Qwen3-VL；必要时显式维护 `cache_position`。
- [ ] 实现 `clone_past(past_key_values)`：
  如果是 `DynamicCache`，优先用官方 clone/crop API；否则逐层 clone tensor。不能只保存引用，因为后续 base generation 可能 mutate cache。
- [ ] 实现 `force_continue(prefix_cache, forced_token_ids, max_new)`：
  先逐 token force，再恢复 greedy decode。
- [ ] 写 `scripts/spike_qwen3vl_inplace_one.py`：
  只跑 1-3 条 FinQA valid target，输出 base text、target token、replay text、injected text、cache diagnostics。
- [ ] 验证 replay control：
  从 `t_star` 强制原始 token后，`base_ans` 和 replay answer 必须一致；若 token-level continuation 不完全一致，记录首个分叉位置。
- [ ] 验证 image KV：
  after prefill 不再传 `pixel_values` / `image_grid_thw`，仍能继续 decode。

**Acceptance**

- [ ] 单样本 spike 能完成 base generation、target detection、prefix-cache clone、force `v_true` replay、force `v_inj` continuation。
- [ ] replay answer 与 base answer 一致；如果 token 不完全一致，有明确、可复现的分叉日志。
- [ ] force `v_inj` 后输出中能看到 injected token 或其合理 continuation，不是重新 prompt。
- [ ] 如果 Qwen3-VL wrapper 无法暴露可用 cache，本阶段停止并写 blocker；不得把 re-prefill fallback 称为 C1-clean in-place。

## 3. v1.5 - TabMWP 主表格算术扩展

### v1.5-1 TabMWP On-Path Locator

**Spec**

v1.5 的目标是把 FinQA 的 clean result 扩展到论文 headline 的 table arithmetic substrate。TabMWP 没有像 FinQA 一样直接可用的 gold program，因此 locator 采用 conservative parse-and-skip：

- 从模型 emitted CoT 中解析最后一个能产生 base/gold answer 的 arithmetic expression。
- 只接受安全表达式：数字、括号、`+ - * /`、百分号 normalization、简单等号链。
- 选择一个 operand，使 `operand -> v_inj` 后表达式重算结果 `c_follow` 与 `c_true` 不同。
- 如果 parse 或 dependency check 不确定，skip，不做 LLM 判定。

**Steps**

- [ ] 实现 `app/eval_distill/onpath_tabmwp.py`。
- [ ] 实现 safe arithmetic parser：
  用 Python `ast` whitelist 或手写 parser；禁止 `eval` 任意字符串。
- [ ] 支持常见 CoT 模式：
  `a + b = c`、`a - b = c`、`a * b = c`、`a / b = c`、`(a+b)/c = d`、`Total = ...`。
- [ ] 从 CoT 后往前扫描 equation candidates，优先 result 匹配 `base_ans/gold` 的表达式。
- [ ] 对 candidate operands 计算 `v_inj` 和 `c_follow`，要求 `c_follow != c_true`。
- [ ] 复用 v1 token-span matching，v1.5 仍默认 single-token；multi-token 放 v2。
- [ ] 添加 TabMWP fixture tests：
  simple addition、subtraction、percentage、ambiguous expression、expression result not answer。
- [ ] dry-run 统计 `data/distill/tabmwp/test_cases_400.jsonl` 上的 parse rate 和 expected valid rate。

**Acceptance**

- [ ] TabMWP locator 宁可 skip，也不接受无法重算 `c_follow` 的样本。
- [ ] 每个 valid TabMWP target 的 `c_follow` 来自 expression recompute，不是 injected operand 本身。
- [ ] dry-run 报告 parse/skip reason，能支撑论文中的 `n_raw -> n_probe` flow。

### v1.5-2 In-Place Controls

**Spec**

v1 先跑最小 corrupt。v1.5 需要补更像 `todo/06.md` 的 matched controls，至少用于 TabMWP 8B，并可回填 FinQA：

Required controls:

- `replay_true`：force original token，sanity only。
- `local_filler`：在 `t_star` force 一个非 numeric filler token；无 target，报告 flip/snap/other。
- `offpath_number`：若同一 CoT 中存在非 target numeric token，替换它；用于估计 numeric disruption 而非 causal operand。

Optional controls:

- `forced_filler_suffix`：从 target sentence anchor 开始，force length-matched filler token sequence，再 continue。
- `forced_shuffle_suffix`：从 target sentence anchor 开始，force shuffled same-token suffix，再 continue。

注意：controls 不报告 follow，因为没有 `c_follow` target。

**Steps**

- [ ] 在 `app/eval_distill/inplace_kv.py` 抽象 `force_sequence_from_anchor(prefix_cache, forced_token_ids, max_new)`。
- [ ] 实现 `local_filler` token policy：
  固定一个 tokenizer 单 token、非 numeric、不会触发 EOS 的 filler；写入 prereg，不按结果调整。
- [ ] 实现 `offpath_number` locator：
  只能选择不属于 parsed expression dependency 的 numeric occurrence；找不到则 skip control。
- [ ] 对每个 valid corrupt target，同步尝试 controls，并在 JSONL 用相同 `sample_id/target_id` 关联。
- [ ] summary 中计算 `corrupt_flip - control_flip`，control 优先级：
  `offpath_number` > `local_filler`；`replay_true` 只做 sanity。
- [ ] 如果 optional suffix controls 显存或稳定性差，记录为 v2，不阻塞 v1.5。

**Acceptance**

- [ ] 每个主 cell 至少有一个 no-target control 的 paired flip readout。
- [ ] replay_true answer match rate 仍接近 1。
- [ ] follow 只出现在 `intervention=corrupt` 行；control 行 readout 仅 `snap/other` 或 `snap/control_flip/other`。

### v2-1 Multi-Token Numeric Support

**Spec**

v1/v1.5 为了降低 cache length mismatch 风险只接受 single-token target。v2 扩展到 multi-token numeric spans：

- `v_true` 可为多 token span。
- `v_inj` 可为不同长度 token span。
- intervention 从 target span 起点的 prefix cache 分支，force `v_inj` token sequence，然后继续 greedy decode。
- 不复用 target 之后的原始 cache，因此 sequence length 变化不是 blocker；但必须正确维护 `cache_position` 和 attention mask。

**Steps**

- [ ] 扩展 `OnPathTarget.token_span = (start, end)`，不再只存 `t_star`。
- [ ] 实现 online span detector：
  用 rolling token ids + decoded normalized text 匹配 target numeric surface。
- [ ] 在 span 起点 clone prefix cache；如果后续发现第二个同值 span，skip ambiguous。
- [ ] `force_continue` 支持多 token forced sequence。
- [ ] 为多 token v_true/v_inj 添加 replay tests：
  force original `v_true` span 后 answer match；force `v_inj` span 后可继续。
- [ ] 将 v1/v1.5 skip 中的 `multi_token_*` 样本重新纳入。

**Acceptance**

- [ ] multi-token replay sanity 通过。
- [ ] 有效样本数显著上升，且 no new prompt / no re-prefill invariant 不变。
- [ ] summary 分别报告 single-token 与 multi-token subsets，避免 reviewer 怀疑混合 artifact。

### v2-2 ChartQA Conservative Locator

**Spec**

ChartQA 是最难的 free-form substrate。v2 不追求高覆盖，而追求 high precision：

- 只接受 CoT 中有可解析 arithmetic expression 的样本。
- expression result 必须匹配 base/gold answer。
- 选择 expression operand 或 intermediate，使 perturbation 后 `c_follow` 可重算且不同于 `c_true`。
- 对纯 visual lookup、yes/no、label answer、无表达式样本全部 skip。

**Steps**

- [ ] 实现 `app/eval_distill/onpath_chartqa.py`，先复用 TabMWP safe parser。
- [ ] 添加 ChartQA-specific normalization：
  percent、axis units、commas、K/M/B suffix 如无法可靠解析则 skip。
- [ ] 不使用 LLM 判断 dependency；如需要人工/LLM audit，只作为 appendix qualitative check，不进入 automatic headline table。
- [ ] dry-run ChartQA 8B/32B base outputs，估算 parseable arithmetic subset。
- [ ] 跑 ChartQA 32B 或 8B present in-place，选择 valid rate 更高且与论文主表更相关的 cell。
- [ ] 跑 same-target two-pass baseline。

**Acceptance**

- [ ] ChartQA valid subset 每条都有 parseable expression 和 `c_follow`。
- [ ] skip_rate 高也可以接受，但必须在论文中写清楚这是 high-precision subset。
- [ ] 不再把旧 ChartQA random corrupt follow 当作 on-path follow。

### v2-3 Full In-Place Control Battery

**Spec**

v2 尝试把 `todo/06.md` 的 controls 全部迁移到 in-place branch：

- `corrupt`：force target value/token sequence。
- `local_filler`：local no-target disruption。
- `offpath_number`：非依赖数字替换。
- `forced_filler_suffix`：length-matched filler suffix。
- `forced_shuffle_suffix`：same-token shuffled suffix。
- `deletion`：从 anchor 后直接不 force 原 suffix，或 force a short bridge token 后 continue；必须标注 length-changing control。
- `paraphrase`：只有在能 tokenize 并 force paraphrased suffix 且数字 multiset guard 通过时才纳入。

**Steps**

- [ ] 定义 target sentence anchor：
  包含 `token_span` 的 sentence/line 起点，所有 suffix controls 从同一 anchor branch。
- [ ] 实现 suffix extraction 与 tokenization，记录 forced sequence 长度。
- [ ] 实现 number-multiset guard，沿用旧 paraphrase cache 但不复用 re-prompt continuation。
- [ ] 对每个 control 写明是否 target-free、是否 length-preserving、是否 semantic-preserving。
- [ ] summary 表按 control family 分组：
  local disruption、numeric offpath、format suffix、semantic paraphrase。

**Acceptance**

- [ ] reviewer 能看到 corrupt 的 effect 不只是 local token disruption 或 shuffled suffix OOD。
- [ ] control 行不报告 follow；只报告 flip/snap/other。
- [ ] full battery 的失败 control 不阻塞核心 corrupt result，但必须在 appendix 透明报告。

## 6. Go / No-Go Gates

### Gate A: KV-cache spike

**Spec**

如果 Qwen3-VL cache branch 无法可靠 replay，不能继续规模跑。

**Steps**

- [ ] 若 replay_true answer match < 0.98，暂停 scale run。
- [ ] 检查 `cache_position`、attention mask、DynamicCache clone/crop、PEFT wrapper 是否改变 generation path。
- [ ] 若只能通过 re-prefill 重建 prefix，标注为 `replay_prefill`，不得称为 C1-clean in-place。

**Acceptance**

- [ ] 只有 C1-clean spike 通过，才进入 v1 scale。

### Gate B: Valid n

**Spec**

v1/v1.5 的目标不是满 400，而是足够窄但可信的 valid subset。

**Steps**

- [ ] 若 valid n >= 150：进入主表。
- [ ] 若 80 <= valid n < 150：只进 appendix 或标注 underpowered，并优先推进 multi-token v2。
- [ ] 若 valid n < 80：不作为 headline causal result，改为 spike evidence。

**Acceptance**

- [ ] 不用低 n 的 null 支撑强 claim。

### Gate C: Divergence

**Spec**

in-place 高 follow 是 thesis-changing，不是坏结果。

**Steps**

- [ ] 如果 in-place follow 明显高于 two-pass：冻结旧 "written chain not load-bearing under both paradigms" 表述。
- [ ] 新叙事改为：prompt-level re-query can destroy stream faithfulness; emitted chain may be load-bearing only in its native autoregressive context。
- [ ] 重新命名 paper title/subtitle，避免 `Reading, Not Reasoning` 过强。

**Acceptance**

- [ ] 结果解释先服从实验，不反向服务标题。

### Gate D: Panel Pitfall Guardrails

**Spec**

每个 v1/v1.5/v2 结果包都必须通过从 `todo/0627.md` 提炼出的审稿护栏。它们不是写作阶段的附加项，而是结果能否进入主文的前置条件。

护栏覆盖：

- 方法透明度：reviewer 不读代码也能重建实验。
- 统计不确定性：不再只报裸比例。
- 样本选择偏差：报告 raw -> eligible -> valid 的流失，并做 sensitivity。
- control 清洁度：至少有一个温和、局部、no-target control，不只依赖 harsh shuffle。
- claim 边界：因果结论只针对 emitted written chain，不否定 latent computation。
- field positioning：明确不是重复 Lanham-style probe，而是 VLM distillation + fixed visual evidence + stream-level intervention。
- 机制语言：`snap` 不自动等于“模型视觉重读”；除非有视觉反事实，否则写成 "answer remains recoverable from visible evidence / original source"。
- 外部效度：若仍只用 Qwen-family，主文明确收窄。
- AAAI 合规：模板、双盲、reproducibility checklist、artifact appendix。

**Steps**

- [ ] 每个 clean probe summary 同时输出 `n_raw -> n_base_correct -> n_cot_present -> n_locator_valid -> n_single_or_multi_token_valid -> n_scored`。
- [ ] 每个 follow/snap/flip rate 带 95% CI；`corrupt_flip - control_flip` 带 paired bootstrap 或 exact randomization CI。
- [ ] 对 multi-cell headline 结果标明 confirmatory vs diagnostic；需要多重比较时用 Holm/BH，或明确不做 confirmatory claim。
- [ ] 对 skipped/failed locator 样本做 sensitivity bound：
  failed parse as non-follow / unknown 的上下界至少在 appendix 报告。
- [ ] 每个主 cell 至少有一个 no-target local/offpath control；若没有，只能进 appendix，不能作为 headline causal result。
- [ ] 对 stable-correct、SFT-gain、SFT-loss、both-wrong 至少在 TabMWP 或 FinQA 主 cell 报一个分层分析；若某层不适用，写明原因。
- [ ] 每个 result snapshot 写明：
  `object audited = emitted written rationale`，`not audited = hidden-state / latent computation`。
- [ ] 文稿中把 TallyQA 标为 external regime check，FinQA 标为 constructive/gold-program operand test，不把异质 substrate 硬堆成同一个因果证明。
- [ ] 文稿中把 `re-read` 降调为 "snap to recoverable source"；只有完成局部遮挡或 counterfactual chart/table editing 后才恢复强 `re-read` 机制语言。
- [ ] 如果没有非 Qwen 复现，Abstract/Discussion/Limitations 明确写 "single model family" 或等价限定。
- [ ] 每次 paper edit 后检查：
  title/abstract/intro/result/discussion 中的 `not reasoning`、`does not compute`、`internalization failed` 等强措辞是否只在限定范围内出现。
- [ ] 在 Method 增加或维护定位表：
  text CoT / visual evidence / tool trace / latent state / chart-table QA；列为 object audited、intervention type、model setting、allowed conclusion。
- [ ] 在 Discussion 增加 deployment audit implications：
  when to show CoT、when to suppress/label CoT、risk disclosure、不能用“不忠实 CoT”当作不解释的借口。
- [ ] AAAI 版本前完成：
  AAAI template/page limit、双盲 metadata/path/link 检查、reproducibility checklist、artifact appendix、最小复现命令。

**Acceptance**

- [ ] 任一结果进入主文前，都有方法透明度、CI、selection flow、control、claim-boundary 五项证据。
- [ ] reviewer 不能用 0627 的 P0-2/P0-3/P0-4/P0-6 任何一条单独击穿主结论。
- [ ] 若某护栏无法满足，结果只能标为 diagnostic/appendix，不能作为 headline causal claim。
